# Firebase 저장 전략: WriteBatch (일괄 처리) 활용

# DB 저장 설계: 효율성과 일관성을 모두 잡는 일괄 처리 전략

> 성능 저하 없이 반복 작업을 정확하게 수행하기
>
> - 앱 '티클릿'의 핵심 기능 중 하나인 '반복 티클 만들기'를 구현하며, 데이터가 하나라도 누락되지 않는 무결성과 사용자에게 빠른 반응을 돌려주는 응답성을 모두 확보하고자 Firebase의 일괄 처리(WriteBatch) 방식을 선택했습니다.

## 1. 과제: 반복 작업에서 데이터의 '완전함' 지키기

### 1.1. 설계 배경 (Context)

티클릿의 [내 티클] 기능은 사용자가 설정한 규칙(템플릿)인 **Task**와 그 규칙에 따라 실제로 만들어진 실행 일정인 **티클(Teekle)** 이라는 두 가지 데이터를 중심으로 돌아갑니다. 시스템은 사용자가 정한 규칙을 해석해 여러 개의 티클을 만들어 Firestore DB에 저장하게 됩니다.

### 1.2. 문제 직면: '모두 저장되거나, 아예 안 되거나' (원자성)

[내 티클]에서 사용자가 반복 설정을 적용하면, 시스템은 계산된 날짜만큼 순회하여 **N개의 Teekle 인스턴스를 생성**합니다. 이때 생성된 Teekle들은 하나의 반복 규칙으로 묶인 하나의 논리적 단위(반복 패턴)를 구성하며, 사용자는 이를 '단 한 번의 생성 버튼 클릭'으로 인식하므로, 내부 작업도 하나의 묶음으로 처리되어야 합니다.

**부분 생성의 위험** : 여러 개의 티클을 하나씩 따로 저장(`add()`와 같은 single writes)하다가 중간에 네트워크가 끊기면, 일부는 저장되고 일부는 누락되는 현상의 가능성이 있습니다. 이는 반복 일정의 규칙을 깨트리는 치명적인 결과로 이어집니다.

**수정 작업의 복잡함** : 반복 설정된 티클을 사용자가 수정하는 경우에도 동일한 원칙이 적용됩니다. 기존에 만들어진 뒤쪽 일정들을 모두 지우고, 새로운 설정으로 다시 만들어 저장해야 합니다. 이 과정은 실제로는 아래와 같은 복합적인 작업으로 처리됩니다.

<aside>

1. 기존 반복 패턴에 속한 **해당 Teekle 이후의 Teekle들을 모두 삭제**
2. 사용자에 의해 수정된 설정값을 기반으로 **N개의 새로운 Teekle을 다시 생성**
3. 이를 다시 Firestore에 **저장**

</aside>

즉, 사용자는 UI에서 “수정하기”로 인지하지만, 실제 내부적으로는 기존 N개의 문서를 삭제하고 새로운 N개를 재생성하는 복합 작업이 수행되는 구조입니다.

반복 패턴은 경우에 따라 매우 많은 수의 Teekle을 생성할 수 있으므로, 생성과 수정 모두에서 **데이터의 일관성과 반복 패턴의 완전성**을 반드시 보장해야 합니다.

이 과정에서 '삭제'와 '생성' 중 어느 하나라도 실패하면 데이터가 뒤엉키게 되고, 사용자에게 잘못된 일정 목록이 보일 가능성이 있습니다. 즉, **N개의 쓰기 작업은 반드시 원자성(Atomicity)을 보장**해야 합니다.

## 2. 구현 전략: WriteBatch(일괄 처리)로 안정성 확보하기

티클을 만들거나 수정할 때는 수십, 수백 개의 데이터를 한꺼번에 다룹니다. 이때 기존 데이터를 읽어와서 비교할 필요 없이 바로 쓰고(create) 지우는(delete) 작업이 주를 이루기 때문에, 파이어베이스가 제공하는 두 가지 방식 중 **일괄 처리(WriteBatch)**를 골랐습니다.

Firestore에서 데이터가 깨지는 것을 방지해주는 `runTransaction`과 `WriteBatch` 중, 프로젝트의 특성을 고려해 `WriteBatch`를 채택했습니다.

| **핵심 목적** | 여러 작업의 **일괄 처리 (Bulk Operations)** | **데이터를 읽고 확인한 뒤 수정** |
| **안정성** | **보장:** 모든 작업이 성공하거나 같이 실패함 | **보장:** 모든 작업이 성공하거나 같이 실패함 |
| **격리성** | **X.** 동시 접근 제어 및 읽기 검증 미제공 | **O.** 동시 변경으로부터 보호 |
| **효율** | **매우 빠름.** 단 1회의 네트워크 왕복(RTT)으로 N개 작업 처리 | **비교적 느림.** 읽기 단계와 재시도 과정 포함 |
| **주요 용도** | 대량의 데이터 생성 및 삭제 | 동시 수정이 잦은 카운터나 잔액 관리 |

### 2.2. WriteBatch(일괄 처리)를 선택한 이유

`WriteBatch`를 선택한 근본적인 이유는 티클릿의 생성/수정 로직은 굳이 복잡한 트랜잭션을 쓸 필요가 없다고 판단했습니다.

1. **비교 과정 불필요**
    - 티클을 만들 때 각 데이터에 고유한 식별 번호(UUID)를 새로 부여하므로, 기존 데이터를 읽어서 충돌 여부를 확인할 필요가 없습니다.
2. **성능 최적화**
    - 수백 개의 티클을 만들더라도 서버와 딱 한 번만 통신하면 됩니다. 덕분에 사용자는 기다리는 시간 없이 쾌적하게 앱을 쓸 수 있으므로, UX에 결정적인 이점을 줍니다.
3. **깔끔한 코드**
    - 실패했을 때 다시 시도하는 복잡한 과정이 없어 코드가 간결하고 읽기 좋습니다.

## 3. 구현 상세 설명

### 3.1. 서비스 계층에서의 활용

`TeekleService`는 여러 개의 티클 목록을 넘겨받아 WriteBatch(일괄 처리) 묶음을 만듭니다. 모든 작업을 목록에 추가한 뒤 마지막에 '커밋(Commit)' 명령을 내려 한 번에 실행합니다.

```dart
// 티클 서비스 클래스
// data/service/teekle_service.dart

class TeekleService {
  // ... (생략)

  /// 여러 티클을 한꺼번에 저장합니다.
  Future<void> createTeekles(List<Teekle> teekles) async {
    try {
      // 1. 일괄 처리(Batch) 시작
      WriteBatch batch = _firestore.batch();

      for (Teekle teekle in teekles) {
        DocumentReference ref = _firestore
            .collection(_collectionName)
            .doc(teekle.teekleId.toString());
        // 2. 일괄 처리 목록에 쓰기 작업 추가
        batch.set(ref, teekle.toMap());
      }

	  // 3. 단 한 번의 요청으로 모든 작업을 안전하게 확정
      await batch.commit();
    } catch (e) {
      // 하나라도 실패하면 전체가 취소되어 데이터 오염을 막습니다.
      throw Exception('Teekle 생성 실패: $e');
    }
  }

  /// 수정 시 특정 날짜 이후의 티클들을 한 번에 지웁니다.
  Future<void> deleteTeeklesFromDateByTaskId({ ... }) async {
    // ... (쿼리 실행)
    WriteBatch batch = _firestore.batch();
    for (var doc in querySnapshot.docs) {
      batch.delete(doc.reference);
    }
    await batch.commit();
  }
}
```

## 4. 맺음말: 이 설계가 주는 가치

`WriteBatch`을 도입하여 티클릿의 데이터 계층을 더욱 튼튼하고 빠르게 만들었습니다.
뿐만 아니라 사용자가 좀 더 나은 경험을 하도록 돕습니다.

- **끊김 없는 반복 경험 (원자성):** 복잡한 반복 생성 및 수정 로직에서도 데이터가 누락되거나 불완전해지지 않음을 보장합니다.
- **빠른 데이터 저장 (성능):** 수백 개의 일정을 순식간에 처리하여 사용자가 막힘없이 다음 화면으로 넘어갈 수 있게 돕습니다.
- **의도가 명확한 도구 선택:** 대량 작업이라는 목적에 딱 맞는 도구를 사용함으로써, 코드가 무엇을 하려는지 명확히 드러나도록 했습니다.
